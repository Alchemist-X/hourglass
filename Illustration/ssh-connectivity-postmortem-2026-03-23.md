# SSH 连通性复盘（2026-03-23）

英文版见 [ssh-connectivity-postmortem-2026-03-23.en.md](ssh-connectivity-postmortem-2026-03-23.en.md)。

最后更新：2026-03-23

## 结论

这次排障的真正分界点，不是“账户有没有配好”，而是“外部 SSH 流量有没有真正进入目标机器”。

最终结论：

- Hostinger 那台机器里，SSH 服务本身是正常的
- 但外部到 Hostinger 的 SSH 流量没有稳定进入虚机
- 阿里云这台机器里，SSH 服务也是正常的
- 改走 `443` 之后，外部流量可以真正进入机器并完成公钥登录

## Hostinger 为什么登不上

已验证的事实：

- 主机内 `sshd` 正常
- 主机自己访问 `127.0.0.1:22`，能收到 SSH banner
- 主机自己访问 `76.13.191.106:22`，也能收到 SSH banner
- `AutoPulse` 用户、公钥、`ufw`、`fail2ban` 都已检查过，不是主因

外部现象：

- 本地连 `76.13.191.106:22` 时，TCP 看起来能建立连接
- 但 SSH 客户端经常卡在 `banner exchange timeout`
- 有时又会在更早阶段被远端直接关闭

最关键证据：

- 在 Hostinger 机器内对外部源 IP 抓包时，`tcpdump` 抓到 `0 packets`
- 同时 `journalctl -u ssh` 里也没有对应外部连接日志

这说明：

- 外部 SSH 流量没有真正进入这台 Ubuntu
- 问题不在 Hostinger 机内 SSH 配置本身
- 更像是在 Hostinger 前面的入口层、转发层或上游网络路径

## 阿里云为什么可以登

阿里云这台机器的基础条件更清楚：

- `sshd` 正常
- 本机 Web Terminal 正常
- 公钥已成功写到 `admin` 用户
- 防火墙规则可控

更重要的是，阿里云最后采用了不同的入口：

- 不再使用 `22`
- 改为 `443`

已验证的事实：

- `47.85.38.253:443` 能正常返回 SSH banner
- SSH 握手能继续走完密钥交换
- 服务器接受公钥
- 最终成功登录 `admin`

成功标志是：

- 已出现 `Authenticated to 47.85.38.253 ([47.85.38.253]:443) using "publickey".`

这说明：

- 不是本地密钥配置错了
- 不是阿里云机器本身有问题
- 关键差别在于：`443` 这条入口路径可用，而 `22` 在当前本地网络 / VPN 路径下不稳定

## 这次真正暴露出的根因

更像是两个因素叠加：

1. 当前本地网络 / VPN 路径对 SSH `22` 很不友好
2. Hostinger 前面的入口层更黑盒，外部流量甚至没有稳定进入虚机

这也解释了为什么：

- Hostinger `22` 不行
- 阿里云 `22` 也不行
- 但阿里云 `443` 可以

## Hostinger 能不能走 443

可以，但这里要分成“理论上可以”和“这次是否已证明可行”两件事。

理论上可以：

- 只要 Hostinger 机器里的 `sshd` 监听 `443`
- 同时主机内和面板层都放行 `443`
- 就可以像阿里云一样走 `SSH over 443`

但这次没有完成 Hostinger 的 `443` 实测，所以不能直接说它一定能通。

原因：

- 当前 Hostinger 的核心问题，不是账户配置
- 而是外部 SSH 流量没有稳定进入虚机
- 如果这个问题发生在 Hostinger 前面的入口层，那么改成 `443` 也可能仍然失败

所以更准确的判断是：

- Hostinger 走 `443` 在技术上可做
- 但是否真的能解决当前问题，需要单独实测
- 不能只靠推断下结论

## 当前建议

如果目标是尽快推进部署：

- 优先继续使用阿里云
- 默认走 `admin@47.85.38.253:443`
- 默认使用本地专用 key `id_ed25519_autopulse`

如果后面还想回头救 Hostinger：

- 通过 Hostinger 控制台进入机器
- 给 `sshd` 增加 `Port 443`
- 放行 `443`
- 再单独做一次外部实测

## 当前可复用结论

- “SSH 是否监听” 不能等于“外部一定能登录”
- 最关键的排障动作，是验证外部流量是否真的进入虚机
- 当前本地网络 / VPN 路径下，`22` 不应再作为默认 SSH 入口
- 对当前环境，更稳的默认方案是 `SSH over 443`
