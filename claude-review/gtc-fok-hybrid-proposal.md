# GTC + FOK 混合下单策略提案

> 状态：提案阶段，不替代当前 FOK 逻辑

## 一、当前现状

所有订单都用 **FOK（Fill or Kill）**：立即全额成交或全部取消。好处是确定性 100%，坏处是每笔都付 Taker fee。

Geopolitics 类市场 fee = 0，所以目前影响不大。但如果扩展到 Politics / Tech / Crypto 类市场，手续费会成为显著成本（往返最高 3.6%）。

## 二、混合策略核心思路

**不是全部替换为 GTC，而是根据场景选择最优下单方式。**

```
每笔订单 → 评估条件 → 选择策略

条件 A：fee = 0（Geopolitics）           → FOK（免费，要确定性）
条件 B：时效性强（新闻驱动、止损）        → FOK（必须立刻成交）
条件 C：不着急 + fee > 0 + spread 合理    → GTC 优先，FOK 兜底
条件 D：流动性极薄                        → FOK 小额试探
```

## 三、GTC 优先策略详细设计

### 3.1 下单流程

```
1. 决策产出 → 需要买入 100 shares @ market ask $0.80
2. 检查 fee：如果 base_fee = 0 → 直接 FOK，结束
3. 检查时效：如果是止损/close/reduce → 直接 FOK，结束
4. 计算 spread：bid $0.78, ask $0.80, spread = 2.5%
5. 挂 GTC 限价单 @ $0.79（mid price，在 spread 中间）
6. 等待最多 T 分钟（建议 T=30）
7. 检查成交状态：
   - 全部成交 → 完成，省了 100% 手续费
   - 部分成交 → 对未成交部分：重新评估，可选择继续等或 FOK 扫尾
   - 完全未成交 → 取消 GTC，改用 FOK 成交
```

### 3.2 GTC 定价策略

| 场景 | 定价 | 理由 |
|------|------|------|
| Spread ≤ 1% | bid + 1 tick | spread 很窄，稍微提价就能排前面 |
| Spread 1-3% | mid price | 在中间等，平衡成交概率和价格改善 |
| Spread > 3% | bid + 30% of spread | spread 太宽说明流动性差，不要太激进 |

### 3.3 超时与兜底

| 参数 | 建议值 | 理由 |
|------|--------|------|
| GTC 等待超时 | 30 分钟 | pulse:live 本身就是定时跑的，30 分钟内能等到 |
| 部分成交阈值 | 已成交 > 60% → 放弃剩余 | 避免为小尾巴付 FOK fee |
| FOK 兜底条件 | 超时 + 未成交 > 40% | 确保仓位不会卡在"半开"状态 |
| 最大重试次数 | 1 次（GTC → FOK） | 不无限循环 |

## 四、适用场景分析

### 4.1 开仓（Open）

| 方面 | GTC 优先 | 纯 FOK |
|------|---------|--------|
| **适合度** | 高 — 开仓不着急，可以等更好的价格 | 中 — 简单但付费 |
| **风险** | 价格跑了（miss trade）| 多付手续费 |
| **省钱** | 进场费 100% 节省 | 0 |

**结论：开仓适合 GTC 优先。**

### 4.2 持仓复审 — Hold / Add-on

| 方面 | GTC 优先 | 纯 FOK |
|------|---------|--------|
| **适合度** | 高 — add-on 完全不着急 | 中 |
| **风险** | 极低 — 加仓错过不影响现有持仓 | 多付费 |

**结论：Add-on 最适合 GTC。**

### 4.3 平仓 / 止损（Close / Reduce）

| 方面 | GTC 优先 | 纯 FOK |
|------|---------|--------|
| **适合度** | 低 — 止损必须立刻执行 | 高 — 确定性第一 |
| **风险** | 挂单等待期间价格继续恶化 | 手续费成本 |

**结论：止损/平仓必须 FOK。**

### 4.4 新闻驱动交易

| 方面 | GTC 优先 | 纯 FOK |
|------|---------|--------|
| **适合度** | 极低 — 新闻窗口稍纵即逝 | 高 |

**结论：新闻交易必须 FOK。**

## 六、预估收益

以 **Politics 市场**为例，假设每月 10 笔开仓交易，每笔 $50：

| 策略 | 每笔进场费 | 月度总费用 | 年度总费用 |
|------|-----------|-----------|-----------|
| 纯 FOK | ~$0.32 | $3.20 | $38.40 |
| GTC 优先（70% 成功率） | ~$0.10 | $0.96 | $11.52 |
| **节省** | | **$2.24/月** | **$26.88/年** |

对当前 ~$500 bankroll，年省 $27 = **5.4% 的年化收益改善**。

如果 bankroll 增长到 $5,000：年省 $269 = 5.4%。

## 八、优缺点总结

### 优点
1. **省手续费** — 非 Geopolitics 市场省 70%+ 的进场费用
2. **可能拿到更好的价格** — 在 spread 中间挂单，比吃 ask 便宜
3. **Maker 返还** — 不仅免费，还能拿到 Taker 费用的 20-50% 返还
4. **对系统架构影响小** — 只改 execution 层，不影响 Pulse/决策/风控

### 缺点
1. **成交不确定** — GTC 可能挂着不成交，错过交易机会
2. **增加系统复杂度** — 需要订单管理、状态轮询、超时处理
3. **"半开"仓位风险** — 部分成交时仓位不完整
4. **不适合定时执行模型** — 当前 pulse:live 是"跑完即退出"，GTC 需要持续监控
5. **回测困难** — 无法准确模拟 GTC 的成交概率

### 最大的架构挑战

当前 `pulse:live` 是 fire-and-forget：跑完 Pulse → 决策 → 下单 → 退出。GTC 需要一个**持续运行的进程**来监控订单状态。这和 stateless 架构矛盾。

两种解决思路：
- **A）两阶段执行**：pulse:live 挂 GTC 单后退出 → 5 分钟后跑一个 `pulse:settle` 检查成交状态 → 未成交的改 FOK
- **B）单次执行内等待**：pulse:live 挂 GTC 后在进程内等 30 分钟，轮询检查，超时后 FOK 兜底

## 九、建议路径

1. **现在不改** — 当前 Geopolitics 免费，没有紧迫性
2. **扩展到收费类别时实施** — 当开始交易 Politics/Tech/Crypto 时再加
3. **先实现两阶段模式（A）** — 对现有架构侵入最小
4. **逐步验证** — 先在 recommend-only 模式下模拟 GTC 定价和成交概率
