## 失败原因排名（按频率）

### 第 1 名：风控护栏静默拒绝（~40% 的被拦截决策）

**表现：** "guardrails removed the open decision"

出现在以下运行中：
- `2026-03-17T024012Z`
- `2026-03-20T070030Z`
- `2026-03-20T073927Z`
- `2026-03-23T075303Z`

**根因：** `applyTradeGuards()` 的级联 `min()` 把订单金额压到 0，但不告诉你是哪个限制卡住了。最常见的约束：
- 总敞口余额耗尽（已有持仓占满 50% 上限）
- 单笔交易上限 5%（$10k bankroll 下最多 $500，小 bankroll 下更低）
- 事件敞口上限 30%

**改善状态：** Agent 1 已完成 `applyTradeGuardsDetailed()` 重构，现在会返回具体是哪个约束。

### 第 2 名：资金不足 / 现金余额为零

**表现：** `cash_balance_usd=0`

**描述：** 多次运行中出现。当所有资金已经部署在持仓中时，没有现金开新仓。加上 50% 总敞口上限，实际可用资金更少。

**典型场景：** bankroll $20，已有 3-4 个小持仓，现金归零，所有新开仓决策被拒。

## 关键洞察

### 最核心的问题：资金太少 + 限制太多 = 几乎无法交易

实际投入只有 **$20-128** 的 equity，在以下限制叠加下：
- 单笔上限 5%（$20 bankroll = 单笔最多 $1）
- 总敞口上限 50%（$20 bankroll = 最多部署 $10）
- Polymarket 最低 5 shares（@ $0.50 ask = 最低 $2.50）
- 内部最低 $10

**$1 的订单无法满足 $2.50 的交易所最低要求。**
**$1 的订单也无法满足 $10 的内部最低要求。**

这就是为什么 75% 的交易都被拦截了。

### 已修复 vs 待修复

| 问题 | 状态 |
|------|------|
| 风控拒绝原因不透明 | 已修复（Agent 1 完成） |
| 预检失败 | 已修复（配置问题，3/16 后未再现） |
| 数据库连接失败 | 切 pulse-live 后自动消失 |
| 风控阈值过于保守 | **待修复**（Phase 1.1 + 1.2） |
| Pulse 抓取不到 5000 市场 | **待修复**（Phase 1.1） |
| 小 bankroll 下交易所最低限额冲突 | **待修复**（需提高 MAX_TRADE_PCT 或降低 MIN_TRADE_USD） |
| 卖出余额不匹配 | **待修复**（需要执行前校验可用余额） |

### 建议优先级

1. **立即：** 提高 `MAX_TRADE_PCT` 到 15-20%（让小 bankroll 也能下单）
2. **立即：** 降低 `MIN_TRADE_USD` 到 1（或直接用交易所最低限额作为唯一下限）
3. **立即：** 降低 `PULSE_MIN_FETCHED_MARKETS` 到 500
4. **短期：** 在卖出前校验链上可用余额（而非仅用远程报告的持仓数）
5. **中期：** 切到纯 pulse-live 路径，消除 DB/Redis 依赖
